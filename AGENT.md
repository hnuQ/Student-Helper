# Agent Notes

- 一般使用权限：在当前项目工作区内进行读取、创建、修改、重命名等常规文件操作时，无需额外询问用户；仅在可能影响系统环境、工作区外目录、网络访问、安装/卸载依赖、删除大量数据或其他高风险操作时再征求确认。
- 反思记录：曾在 Windows 终端环境下批量改写文件时，把“终端显示乱码”和“文件内容真实乱码”混为一谈，又直接覆盖了含中文文案的源码，导致界面出现 `??` 和乱码文本。这类问题属于编码处理失误，不能再重复。
- 文案修改约束：凡是涉及中文文案、菜单、标题、占位符、提示语的批量修改，禁止仅凭终端输出判断编码状态；修改前后都必须执行一次面向源码的可见文案扫描，确认没有新增 `??`、乱码汉字或被污染的中文字符串后再交付。
- 桌面应用开发经验：本地持久化功能不能只看“点击后界面有没有反应”，必须验证完整链路：渲染层提交、主进程写库、重新读取、切换页面后回显、重启应用后仍然存在。任何一个环节没验证，都不能算完成。
- 桌面应用开发经验：涉及“按天记录”的字段时，优先使用稳定的日期键（如 `yyyy-MM-dd` 字符串）而不是直接用 `DateTime` 做唯一键，除非已经明确处理了时区、序列化和数据库比较规则。对日历、周报、按天查询这类功能，字符串日期通常更稳。
- 桌面应用开发经验：UI 结构要和用户操作模型一致。如果用户预期是“先看日历，再点某一天进入详情”，就不要默认展开当天表单。交互模型错了，即使功能能用，用户也会觉得设计有问题。
- 桌面应用开发经验：日历卡片上的摘要信息必须对应明确字段，不能靠模糊推断。像“学习时长”“训练计划”这种需要直接展示的内容，应在数据模型里有稳定字段，而不是临时从自由文本里猜。
- 桌面应用开发经验：每次改动桌面应用中的中文菜单、窗口标题、导航和关键页面后，要至少做一次“可见区域复查”，覆盖左侧导航、顶部标题、对话框按钮、表单标签、菜单栏和空状态文案，避免用户一眼就看到未清理干净的错误文本。
